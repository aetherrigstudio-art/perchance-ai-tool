### OPEN — Finding A: `prepUserInput()` applied inconsistently (MEDIUM)
- **char-wiz-html:728** — `"USER'S IDEA:\n" + val("scenarioNotes")` (scenario gen) → **raw**
- **char-wiz-html:757** — `"FOCUS …:\n" + focus`, `focus = val("loreNotes")` (lore gen)
  → **raw** (reused raw at 753/769)
- Contrast char-wiz-html:562/582 (character/persona) which correctly wrap in
  `prepUserInput(...)`.
- **Impact (bounded):** single-user client-side tool; the realistic threat is a user
  corrupting their *own* generation or leaking prompt scaffold via injected
  `=== SECTION ===` headers — not cross-user compromise. But it's a clear, avoidable
  inconsistency in an existing control.
- **Fix:** wrap both in `prepUserInput()` for parity (`val("scenarioNotes")` →
  `prepUserInput(val("scenarioNotes"))`; same for the `focus`/`loreNotes` uses).
  One-line each; re-run `node test/smoke.mjs`. Requires a `char-wiz-html` edit →
  per workspace rule 7, plan via `/plan` first.

### OPEN — Finding B: sister tools have no input sanitizer (LOW)
`prepUserInput()` lives only in `char-wiz-html`. `fixer-html-panel-1.txt` (`brokenText`)
and `image-style-builder-html-panel-8.txt` (`styleNotes`) inject paste/notes raw
(prior audit F9). Low risk (single-user, the pasted character is the subject of the
fix), but no parity with the wizard's hardening.

### Defense-in-depth note: `buildSceneCode` placeholder collision (already in ISSUES.md)
`buildSceneCode` does sequential `.replace("__WARDROBE__"/"__STYLEPREFIX__"/…)`. A
character field containing a literal `__STYLEPREFIX__` can make a later `.replace`
substitute inside the already-inserted wardrobe JSON, corrupting the generated
`customCode`. **Low XSS exploitability** (ACC runs arbitrary `customCode` by design,
so this is bounded by ACC's own trust model), but a real **robustness** bug — already
tracked in `ISSUES.md` MEDIUM (~line 1056/1545). Fix: unique sentinel delimiters or a
single structured substitution.
